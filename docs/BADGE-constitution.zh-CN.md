# BADGE 开发宪法 v1.2

> **B**oundary **A**nd **D**efensive **G**uard for **E**ngineering — 边界与守护工程开发宪法
>
> 好项目不是靠勤奋检查出来的，是靠设计让人根本没法犯错。
> 本宪法是工程品味的沉淀——它告诉你为什么这么设计是对的，不只是告诉你应该怎么做。

# 第一层：设计哲学

## 一、边界思维

边界是系统设计的第一步。大到服务之间的网络边界，小到函数之间的参数边界，边界划在哪里，决定了系统的复杂度分布、可测试性、可扩展性。先划边界，再填实现。

### 1.1 模块边界：一事一责

每个模块只做一件事，把这件事做好。如果一个模块的职责无法用一句话说清楚，就该拆。模块内部可以很复杂，但模块之间的边界必须清晰简单。

**边界好的实际效果：** 修改上层代码时，不需要打开下层文件。比如一个 agent 主模块调用了十几个工具，你只需要知道每个工具的接口和作用，不需要看它们的实现代码。哪怕这些工具加起来有几千行，都和你修改 agent 调度逻辑没关系。这就是边界的力量——它把修改的影响范围限定在边界之内，让你可以安全地修改，不用担心中下游的连锁反应。

**判断标准：** 你能在不看代码的情况下，仅凭模块名和接口签名，准确说出它的职责吗？如果不行，边界没划清楚。修改一个模块时，你需要打开的下游文件数量是否趋近于零？如果不是，边界还需要加固。

### 1.2 接口边界：稳定抽象，可换实现

模块之间通过稳定的抽象接口通信，不依赖对方的具体实现。新增一个模型后端、新增一个采样策略、新增一个存储引擎——只需实现接口并注册，不修改任何现有代码。对扩展开放，对修改关闭。

**判断标准：** 新增一个实现需要改动几行现有代码？如果超过 0 行，接口边界需要重新设计。

### 1.3 层次边界：计算与设施分离

代码只有两种：一种是做**计算**的（算法、业务规则、数据变换），一种是做**设施**的（GPU 通信、网络 I/O、文件系统、数据库）。计算代码不依赖任何设施，可以在单线程本地跑、可以单独测试、可以在任何环境复现。设施代码负责把计算结果接入真实世界。

这个思想来自 Flink、Hadoop 等大数据分布式系统——不要让工程师一边操心业务逻辑，一边操心底层实现，一边操心基础设施。谁也不可能同时干好这三件事。

**判断标准：** 你能在不启动 GPU、不连网络、不读文件的情况下运行核心计算逻辑的测试吗？如果不行，层次没分离。

### 1.4 数据边界：单一真相源

每一条信息在系统中只有**一个**权威来源。配置只从一个文件读入，不搞环境变量 fallback 链。状态只在一个地方维护，不搞多副本同步。数据只沿一个方向流动，不搞回调、不搞循环依赖。

"这个值可以从三个地方来"听起来灵活，实际上用户改了参数发现没生效，debug 一下午。灵活性在这里不是优点，是 bug 的温床。

**判断标准：** 如果有人问"这个值是从哪来的"，你能在一秒内给出唯一答案吗？如果不能，源头太多了。

## 二、防呆设计

汽车设计有一个原则：踩死油门有转速限制，挂档有同步器，刹车优先于油门，车窗除雾按钮放在容易够到的地方。说明书几百页，没几个车主看过——但事故率很低，因为**系统不允许人犯错**。

软件也应该是这样。不是祈祷开发者小心、祈祷用户仔细读文档，而是用设计让错误根本不可能发生。

### 2.1 契约即防呆

所有边界上的交互都用**可验证的契约**来约束，不用约定和默契。pydantic 模型校验配置字段，ABC 抽象类强制子类实现钩子，类型标注让 mypy 检查数据流——这些不是"最佳实践"，是"防呆装置"。写代码的人不会出错，不是因为他认真，而是系统不允许他出错。

### 2.2 显式即防呆

代码里没有魔法。状态是显式的，数据流是可见的，依赖是声明的。不以字符串拼路径，不以 `**kwargs` 一路透传导致无法追踪参数来源，不以 `hasattr` 探测业务接口。隐式行为是理解代码的最大障碍——对人和 AI 都是。当一切显式，错误就无处藏身。

**`**kwargs` 的合理使用：** 在 ABC 抽象方法的适配器模式中，`**kwargs` 作为扩展点是允许的——不同子类可能需要不同的领域参数，基类不应该为每个子类的特殊需求而修改接口。关键是：`**kwargs` 必须在 docstring 中说明子类可能接受的额外参数，让调用者知道该传什么。HuggingFace 的 `model.generate(**kwargs)` 就是这种模式的标准范例。

**`hasattr` 的合理使用：** 以下三种场景允许使用 `hasattr`：
1. **运行时能力检测**：检查 PyTorch 版本是否支持某个特性（如 `hasattr(torch, "float8_e8m0fnu")`），这比解析版本号更可靠
2. **跨版本兼容性处理**：处理上游库（如 HuggingFace）不同版本返回不同类型的输出（`output.logits` vs `output["logits"]`）
3. **Duck-typing 类型判断**：在不引入硬依赖的前提下判断对象类型（如区分 tensor 和普通数值）

禁止的是用 `hasattr` 探测自己项目的业务接口——那意味着接口契约不清晰，应该用 ABC 或 Protocol 定义明确的接口。

### 2.3 边界校验即防呆

数据在跨边界时必须校验。配置加载时校验所有字段，拒绝未知字段，检查必填项。API 请求发出前校验格式，收到响应后校验结构。输出落盘前检查是否覆盖已有数据、是否泄露敏感信息。错误在边界上就被拦截，不让非法数据流入内部，不让敏感数据流出外部。

**一个核心区别：** 边界校验拒绝非法请求（如配置缺失），这不是"退路"，这是"防线"。防线的作用是阻断，退路的作用是绕行。不要混淆。

## 三、退路设计

退路是一条核心原则：**降级可以改变代价，不能改变结果。**

### 3.1 同效退路

系统可以自动执行，无需告知用户。结果不变，只改变代价——时间、资源、速度。

- 重试：多花了时间，最终拿到的结果是一样的。rollout 失败重试 5 次，全部失败才报错。
- 降速：网络拥塞时降低并发，虽然慢了，但数据完整到达。
- 备胎：限速 80km/h，但能把你送到目的地。

### 3.2 透明退路

结果可能受影响，系统必须明确告知用户。日志 WARNING 级别记录，metrics 暴露，让用户自己判断是否需要干预。

- 跳过损坏样本：日志记录样本 ID 和跳过原因，训练继续，但用户知道发生了什么。
- 使用缓存：数据过期但网络不可达时，使用过期缓存并告知用户。

### 3.3 预授权退路

结果会显著改变，用户必须在配置中**提前声明接受**，默认关闭。这不是运行时决策，是部署前决策。

符合此原则的预授权退路示例：
- 数据加载时跳过损坏的训练样本——仅当用户显式设置配置开关（默认关闭）时才可接受，因为模型将在与预期不同的数据集上训练。
- 覆盖已有输出目录——仅当用户显式选择接受（默认关闭）时才可接受，因为这会破坏之前的运行结果。

宪法不规定具体的配置键名——每个项目定义自己的开关。原则是：如果结果显著改变，用户必须在部署前说"可以"，而非事后才发现。

### 3.4 退路与防线的区别

密码过期不能登录？这不是退路，这是**防线**在起作用——身份验证的边界校验发现不合规，拒绝通过。配置字段缺失退出？这是防线。静默降级到更差的模型？这是**坏的退路**——它改变了结果，而且用户不知道。

## 四、零步上手

一个新用户 clone 项目后，两条命令就能看到产出结果。不需要额外安装系统依赖（都写在 Docker 里），不需要手动配置（默认值就是生产级合理值），不需要理解内部架构（example config 复制一份就能用）。

**零步不是说零配置，而是说默认配置就是可用配置。** 如果用户必须理解 10 个参数的含义才能开始，那就是零步的失败。

**判断标准：** 一个新用户从 clone 到看到第一个产出结果，需要几步操作？超过 3 步就该反思。

---

## 五、先正确，后可优化

性能优化是必要的，但不能以牺牲正确性和可维护性为代价。原则是：**先保证正确，再追求性能。但不要设计成无法优化的架构。**

计算与设施分离已经为性能优化提供了天然的基础——核心算法可以独立调优，不耦合到分布式通信或 I/O 中。接口的稳定抽象保证了替换低效实现时不影响上层。如果为了"代码优雅"而把多个本该独立执行的操作耦合在一起，导致后面想并行化时发现根本拆不开，这就是过度设计——它牺牲了优化空间，却什么都没换来。

**判断标准：** 如果发现某个组件是性能瓶颈，你能在不改动上层代码的情况下替换它吗？如果不能，架构需要重新审视。

## 六、环境可复现

项目在任何两台硬件相同的机器上运行，产出应该**比特级一致**。这不是理想，是可验证的工程标准。如果两次运行结果不同，要么是随机种子没固定，要么是依赖版本不一致，要么是代码里有非确定性操作——这些都应该被消除。

**实现要点：**
- 随机种子在配置文件中显式设定，不依赖系统时间或硬件状态
- `uv.lock` 锁定所有依赖的精确版本，提交到 git
- Docker base image 锁定具体 SHA256 digest（不是 tag，tag 可以被覆盖）
- 每次运行的配置自动备份到输出目录，确保事后可复现

**判断标准：** 两台机器、同一 config、同一 seed，产出是否一致？运行两次，产出是否一致？

# 第二层：工程规范

## 七、配置系统

### 7.1 单一 YAML 入口

所有配置在一个 YAML 文件中，按功能域分段。不使用环境变量（NVIDIA 的 `CUDA_VISIBLE_DEVICES`、`NCCL_*` 已经够乱了，不要再往里面加），不搞 CLI 参数覆盖配置文件的隐式优先级。配置就是配置，环境就是环境，泾渭分明。

### 7.2 加载即校验

配置加载时完成全部校验，不在使用点零星校验。pydantic 的 `extra="forbid"` 拒绝拼写错误的字段名，`model_validate()` 一次性检查所有字段的类型和约束。校验失败立即报错退出，给出明确的错误信息——哪个字段、期望什么、实际收到了什么。

### 7.3 模板即文档

项目提供 `config_example.yaml`，包含所有字段、注释和默认值。用户复制一份改几个值就能用。模板文件本身是一个可用的配置——注释充分、默认值合理，覆盖 80% 的使用场景。

## 八、代码组织

### 8.1 目录布局

```
project/
├── src/project_name/        # 源码，src-layout
│   ├── __init__.py          # 版本号 + 公开 API re-export
│   ├── __main__.py          # python -m 入口
│   ├── py.typed             # PEP 561 标记
│   ├── cli.py               # CLI 入口
│   ├── config.py            # 配置模型定义与加载
│   ├── core/                # 计算层：纯逻辑，零设施依赖
│   ├── backends/            # 设施层：分布式、GPU、网络
│   └── domain/              # 领域逻辑层
├── configs/                 # 示例配置
├── docs/                    # 架构文档
├── tests/                   # 测试（目录树镜像源码）
│   ├── core/
│   ├── backends/
│   └── e2e/                 # 端到端测试
├── scripts/                 # 一次性工具脚本
├── docker/                  # Docker 构建
├── pyproject.toml
├── uv.lock
├── .python-version
├── README.md
├── README.zh-CN.md
├── LICENSE
└── .gitignore
```

### 8.2 文件粒度：既不大也不碎

一个文件大到几千行，阅读和修改都困难，IDE 打开都慢。拆得太碎，一个功能散落在十几个文件里，读者到处跳转，思维被割裂。理想的粒度是：**一个文件对应一个明确的概念单元**，读者打开这个文件就能完整理解这个概念。

判断标准：你能在一个文件里完整理解一个概念吗？如果需要在多个文件间反复跳转才能拼出全貌，太碎了。**建议文件不超过 500 行**——如果超过，问问自己它是不是塞进了两个概念。但如果文件内的逻辑确实属于同一概念单元（如纯函数工具集、复杂模型适配器），超过 500 行是可以接受的；此时应确保文件内部按功能域用清晰的分隔线和注释组织，让读者能快速定位。

### 8.3 类改目录：当类太大时

当一个类的方法多到让单个文件膨胀到几百行甚至上千行，不应该把方法硬塞进一个文件，而应该把类升级为一个目录：

```
backends/models/qwen35_36/
├── __init__.py          # 从 adapter.py 重导出 Qwen35Adapter
├── adapter.py           # 主类定义 + 模板方法骨架
├── forward.py           # 前向传播相关的方法
├── generation.py        # 生成/采样相关的方法
├── checkpoint.py        # checkpoint 存取相关的方法
└── helpers.py           # 纯函数工具（不依赖 self 状态）
```

**原则：** `adapter.py` 保留类的骨架——`__init__`、模板方法、抽象方法。按功能域拆出去的每个文件，包含该类的一组相关方法。外部使用者只 import 类名，完全不感知内部是单文件还是目录。`__init__.py` 负责这个"对外透明"。

内部方法之间如果需要共享状态，通过 `self` 访问——它们仍然是同一个类的方法，只是物理上分散在不同的文件中。不需要 `self` 状态的纯函数放进 `helpers.py`，它们是独立的、可单独测试的。

### 8.4 OOP 与 FP 各司其职

面向对象和函数式编程争论了几十年谁更好。答案是：**它们不是对手，是工具。** 关键是让它们各自做自己擅长的事。

**OOP 擅长：** 有状态的、有生命周期的、有多套实现需要统一接口的场景。类继承表达"同一类事物，不同实现方式"——模型后端、采样策略、调度策略。模板方法让基类控制流程，子类填充差异。

**FP 擅长：** 无状态的、纯计算的、数据变换的场景。函数接收输入，返回输出，无副作用。奖励计算、数据清洗、格式转换、文本解析——这些用纯函数最自然，输入输出一目了然，测试不依赖任何上下文。

**不要为了 OOP 而 OOP：** 如果一段逻辑不需要 `self` 状态，不要强行放进类里。一个独立的纯函数比一个只有一个方法的类更清晰、更容易测试、更容易复用。

**不要在同一个模块里混用两种风格：** 一个文件要么以类为主（有状态管理），要么以函数为主（纯计算管道）。混在一起会让读者困惑：这个状态被谁修改了？这个函数有副作用吗？

### 8.5 输出目录隔离

源码和数据是资产，运行产物是消耗品。两者必须物理隔离——输出目录不在源码树内，由配置文件的 `output` 字段指定。默认值指向项目根目录下的 `outputs/`，这个目录在 `.gitignore` 中排除。

```
outputs/<run_name>/
├── config.yaml           # 本次运行的配置备份（完整复现）
├── checkpoints/          # 模型 checkpoint
├── logs/                 # 按功能域分文件的日志
└── results/              # 最终产出（数据集、模型、评测结果）
```

每次运行自动生成唯一的 `run_name`（默认基于时间戳），防止误覆盖。运行结束后，用户可以根据 `outputs/` 下的 `config.yaml` 完整复现本次运行——不需要回头找原始配置文件。

### 8.6 导入与文件头

统一使用绝对导入，禁止相对导入。循环导入用 `TYPE_CHECKING` 守卫。`__init__.py` 负责 re-export 公开 API，让外部使用者不需要关心内部文件结构。

`from __future__ import annotations`（PEP 563）仅在出现循环导入导致类型注解无法求值时使用，不要求逐文件添加。在 Python 3.11+ 项目中，`str | None` 语法已原生支持，`TYPE_CHECKING` 守卫已能处理大多数循环导入场景，无需在每份文件顶部添加样板代码。

### 8.7 版本号：单一来源

项目的版本号**只在 `pyproject.toml` 的 `[project] version` 字段中维护**。这是宪法 1.4（单一真相源）和 2.1（防呆设计）的直接体现：

- **禁止**在 `__init__.py` 中定义 `__version__` —— 它必然和 `pyproject.toml` 不同步，事实上有两个真相源
- **禁止**在 `config_example.yaml` 注释中写版本号 —— 模板文件是给用户看的，不是版本记录
- **禁止**在代码中硬编码版本号字符串

如果代码运行时需要版本号，从 `pyproject.toml` 读取或使用 `importlib.metadata.version("package-name")`。如果用户需要知道版本，告诉他看 `pyproject.toml` 或运行 `pip show`。

**判断标准：** 改版本号需要改几个文件？如果超过 1 个，违反本条款。

## 九、数据与接口

### 9.1 数据模型：pydantic

所有核心数据结构使用 pydantic `BaseModel` 定义。禁止裸 `dict` 或 `list` 传递业务数据。`extra = "forbid"` 拒绝未知字段。嵌套结构用嵌套 pydantic 模型，不做 dict 套 dict。

`frozen = True` 建议用于从外部数据源加载后不再被修改的纯数据对象（如训练样本、评分结果），它可以防止意外修改。但对于配置对象等需要在初始化阶段被 CLI 参数覆盖的场景，允许不使用 `frozen`；此时应确保初始化完成后不再修改配置对象。

### 9.2 类继承：ABC 模板方法

有多套实现时，ABC 基类定义流程骨架（公开方法），每一步调用 `_` 前缀的私有钩子。具体子类只覆盖钩子，不覆盖公开方法。基类提供钩子的默认实现，减少子类重复代码。扩展新实现时，定义新类并注册，不修改任何现有代码。

### 9.3 注册表：动态扩展

多个同类型实现通过注册表（字符串到类的映射）管理。运行时按名称查找并实例化。查找失败时给出明确的错误信息，列出所有可用选项。新增实现只需一行注册代码。

## 十、CLI 设计

### 10.1 单一命令 + 配置驱动

每个项目只有一个 CLI 命令，只接受一个 `--config` 参数。所有行为由配置文件驱动，CLI 不做逻辑判断。CLI 的职责是：解析参数 → 加载配置 → 执行。仅此而已。

分布式训练场景下，`--gpus` 和 `--master_port` 等少量基础设施参数属于例外，因为它们本质上是运行时环境信息，不是业务配置。

### 10.2 CLI 与 scripts 的边界

**CLI 是面向用户的稳定接口，scripts 是面向开发者的临时工具。** CLI 的参数和输出格式是契约，不能随意改。scripts 是一次性的、实验性的、随时可以删的，不需要考虑向后兼容。如果某个脚本被反复使用、使用范围超出了开发者本人，就该升级为 CLI 子命令。

## 十一、测试

### 11.1 工具链

pytest 运行测试，ruff 格式化 + lint（`ruff format` + `ruff check --fix`），mypy 类型检查。三者配置在 `pyproject.toml` 中，版本锁定在 dev 依赖中。

### 11.2 目录结构

`tests/` 的目录树镜像 `src/` 的目录树。找测试文件不需要思考：`src/project/core/schema.py` 的测试就在 `tests/core/test_schema.py`。端到端测试放在 `tests/e2e/` 独立目录下。

### 11.3 测试规范

单元测试覆盖公开接口，外部依赖用 mock 替代。端到端测试覆盖完整 pipeline。参数化测试覆盖边界条件。测试函数命名描述三个要素：做了什么、什么条件下、期望什么结果。每个测试独立，不依赖执行顺序。

## 十二、代码风格

### 12.1 类型标注

联合类型用 `str | None`，不用 `Optional[str]`。`py.typed` 标记文件。行宽 100，目标 Python 3.11+。`from __future__ import annotations` 仅在出现循环导入导致类型注解无法求值时使用（见 8.6）。

### 12.2 命名

类 PascalCase，函数和变量 snake_case，私有成员 `_` 前缀，常量 UPPER_SNAKE_CASE，类型别名 PascalCase，模块文件 snake_case。

### 12.3 语言规范

`README.md` 和 `README.zh-CN.md` 使用中英双语——项目的门面要同时覆盖英文和中文两大开发社区，这是当前开源世界最活跃的两个语系。

所有其他文件——注释、docstring、架构文档、配置注释——**中文或英文任选其一，作者自行决定**。不允许使用其他语言。单文件内保持语言一致。

**配置模板注释**（`config_example.yaml`）默认使用**英文**——配置模板是面向用户的界面，英文是全球开发者社区的通用语言。

**提交信息**使用英文（见 17.3）。

**技术术语、算法名称、框架名称、学术概念**保留英文原文。例如 Flink、1F1B、KV cache、attention mask、backpressure —— 这些术语的英文原文是领域内工程师的共同语言，强行翻译反而损失信息密度。原则是：**以准确性优先，不以语言纯洁性牺牲技术表达。**

注释写 **why**，不写 **what**。禁止无用注释。

## 十三、错误处理与日志

### 13.1 异常：不吞

**绝对禁止** `except Exception: pass` 或裸 `except: pass`。每一个 `except` 块必须包含明确的处理逻辑（重试、降级、转换为用户可读信息后退出）。异常原则上在边界层统一处理，在内部层尽早暴露。

### 13.2 日志分级

DEBUG 给开发者排查 bug，INFO 给用户了解运行状态，WARNING 给用户评估运行质量，ERROR 告诉用户为什么失败。

### 13.3 日志分文件

日志按功能域分别落盘，每个功能域区分**易读日志**（人类可读，INFO 级别）和**原始日志**（机器可解析，DEBUG 全量）。ERROR 汇聚到公共错误日志文件。所有日志文件位于输出目录下的 `logs/` 子目录中（见 8.5 输出目录隔离）。

具体的文件名和功能域划分由各项目根据自身需求决定——宪法规定的是概念原则，不是文件列表。例如，训练项目可能按训练事件、rollout 生成、奖励计算等分文件；数据管道可能按抽取、转换、加载等分文件。核心不变的是：**每个功能域一个日志文件，每个域区分易读和原始日志，ERROR 汇聚到公共文件。**

## 十四、依赖与部署

### 14.1 许可证约束

所有依赖包必须是 MIT、Apache 2.0 或等价的可商用宽松许可证。禁止 GPL、AGPL 等传染性许可证。引入新依赖前检查许可证兼容性。

### 14.2 依赖管理：uv

所有项目使用 `uv` 作为唯一包管理器。`.python-version` 锁定 Python 版本，`uv.lock` 锁定依赖精确版本，两者提交到 git。`uv sync` 一键搭建开发环境。

### 14.3 Docker 必须支持

每个项目必须提供 Docker 部署。Base image 锁定具体版本（不用 `latest`）。Dockerfile 两层构建（依赖层 + 源码层），利用层缓存。`build.sh` 封装构建命令。

## 十六、版本演进与债务管理

技术负债是工程质量的癌症。今天偷的懒，明天变成两倍工作量，后天变成谁也不敢碰的禁区。核心原则只有一条：**新架构验证通过之后，旧架构必须根除。**

### 16.1 不留负债

- 新架构上线、测试通过、稳定运行后，旧代码**立即删除**，不保留"兼容模式"，不保留 `legacy/` 目录，不保留 `# TODO: remove after v2` 注释。代码库中只存在一套当前架构。
- 删除旧代码的同时，同步更新：命名（不再有 `v2`、`new`、`legacy` 前后缀）、配置模板（`config_example.yaml` 反映当前架构）、文档（README 和 docs/ 中的架构描述）、测试（删除旧架构的测试，不保留"以防万一"）。
- 版本号在每次发布时更新，遵循 `MAJOR.MINOR.PATCH`：架构重构、不兼容的配置变更升 MAJOR，新增功能升 MINOR，bug 修复升 PATCH。

### 16.2 旧系统迁移

当旧版本用户需要升级到新架构时，提供一键迁移方案：

- 迁移脚本放在 `scripts/` 目录下，命名格式：`migrate_v1_to_v2.py`、`migrate_config_v2_to_v3.sh`
- 脚本头部注释注明：源版本号 → 目标版本号、迁移了什么（配置文件格式、checkpoint 格式、数据格式）、不可自动迁移的部分（需要用户手动处理的）
- 迁移脚本是临时工具——只服务于当前这次大版本升级，下一个大版本时可以被新的迁移脚本替代。旧迁移脚本保留在 `scripts/` 中供历史参考，因为可能有用户跳版本升级

**判断标准：** 一个新人 clone 项目后，能看到旧架构的痕迹吗？如果能——遗留代码、过时注释、未更新的配置模板——版本管理就是失败的。

## 十七、开源管理

### 17.1 提交前检查

每次提交前自查：
- 是否有 API key、密码、token、私钥等机密信息？
- 是否有训练数据、用户数据、内部 URL、IP 地址等隐私数据？
- 是否有 `.env` 文件？（只提交 `.env-example`）
- 是否有构建产物、缓存、虚拟环境？（`__pycache__`、`.pytest_cache`、`.venv`、`outputs/`）
- 是否有 IDE 配置文件？（`.idea/`、`.vscode/`）
- 是否有 AI 助手文档？（`CLAUDE.md`、`AGENTS.md`）

### 17.2 开源前最终检查

- 确认 `git log --all --full-history` 中没有敏感文件的历史记录
- 如果曾提交过敏感信息，使用 `BFG Repo-Cleaner` 彻底清除，轮换所有已泄露的密钥
- 确认 `LICENSE` 文件存在且正确
- 确认 README 中没有引用内部资源

### 17.3 Git 工作流

- 禁止直接提交到 `main` / `master` 分支
- 所有开发在功能分支上：`feature/<描述>`、`fix/<描述>`、`docs/<描述>`
- 合并到 main 必须通过 PR，至少自己 review 一遍 diff
- 提交信息用英文，格式：`type: short description`（feat、fix、docs、refactor、test、chore）。英文是开源社区的事实标准，`git log --oneline` 工具链以英文为第一语言，便于国际贡献者理解。
- 一个提交只做一件事

### 17.4 .gitignore 必须覆盖

- Python 运行时：`__pycache__/`、`*.pyc`
- 虚拟环境：`.venv/`、`venv/`
- 测试与类型检查缓存：`.pytest_cache/`、`.mypy_cache/`、`.ruff_cache/`
- 构建产物：`dist/`、`build/`、`*.egg-info/`
- 环境配置：`.env`（保留 `.env-example`）
- 输出与数据：`outputs/`、`data/`（示例数据除外）
- IDE：`.idea/`、`.vscode/`
- AI 助手：`CLAUDE.md`、`AGENTS.md`、`CLAUDE.zh-CN.md`
- 系统文件：`.DS_Store`、`Thumbs.db`

### 17.5 开源协议

默认 MIT 协议。`LICENSE` 文件放在根目录。如果项目依赖了 Apache 2.0 协议的库，考虑使用 Apache 2.0 保持兼容。
